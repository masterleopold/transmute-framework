# Specification Validation -- Detailed Guide

## Role

This guide drives Stage 2B of the Transmute pipeline: independently cross-validating the BRD and PRD for consistency, completeness, and correctness before they propagate into code.

## Stage 2B: BRD ↔ PRD Cross-Validation

You are a senior quality assurance analyst acting as the TEAM LEAD for a multi-agent specification validation project using Claude Code Agent Teams. Your task is to independently cross-validate the BRD and PRD for consistency, completeness, and correctness — catching errors before they propagate into code.

**Stage Sequence**: Business Plan → 0 (Tech Stack) → 1 (BRD) → 2 (PRD) → **2B (this stage)** → 3+4 (Scaffold + CLAUDE.md) → 5 (Implementation) → 5B (Audit) → 6A/6B/6C (parallel) → 6E → 6F → 6G → 6D → 6H → 6V → 6R → 6P/6P-R → 7 (Deploy) → 7V → 7D → 8 (Feedback) / 9 (Maintenance)

## Why This Stage Exists

The BRD and PRD are generated by separate Agent Team sessions. While the PRD session reads the BRD, several categories of errors can slip through:
- BRD requirements that were overlooked or misinterpreted during PRD generation
- PRD specifications that contradict BRD constraints
- Terminology inconsistencies between the two documents
- Missing user stories for functional requirements
- Screen specifications that don't match user flow descriptions
- API specifications that don't cover all screen data needs
- Acceptance criteria that are untestable or ambiguous
- Cross-feature interactions documented in BRD but missing from PRD

Fixing these errors now costs minutes of spec editing. Fixing them after code is written costs hours of rework.

## Known Failure Patterns

Based on observed Plan Cast outcomes, these are common spec validation failures:

1. **Terminology drift**: BRD uses "organization" while PRD uses "workspace" for the same concept. This happens because BRD and PRD are generated in separate sessions with different context.
2. **Coverage gaps from output token limits**: BRD specifies 20 FRs but PRD only has user stories for 15 — the PRD generation agent hit the output token limit (see tech-stack.md § Model Specifications) before covering all requirements. With the pipeline model's context window, input is no longer the bottleneck — but output limits per agent response still cause truncation on large files.
3. **Cross-reference broken links**: PRD screen specs reference API endpoints that don't exist in the API spec file, or user flows reference screens by informal names that don't match SC-IDs.
4. **Quantified-to-vague regression**: BRD NFRs specify "99.9% uptime" but PRD non-functional specs say "high availability" — precision was lost in translation. See the Quantification Rule in the Critical Rules section below.

5. **Untestable acceptance criteria**: Criteria that follow Given/When/Then syntax but are semantically untestable ("Given a user is logged in, When they use the feature, Then it works correctly").
6. **Field name mismatches**: Data model has `organizationId` but API specs return `orgId` — these silent mismatches become runtime bugs in Stage 5.
7. **Mermaid diagram syntax errors**: Diagrams that were valid in BRD but became invalid after PRD edits. Always validate syntax after editing.
8. **Coverage gaps from feature decomposition**: BRD specifies N features but PRD only covers N-5 — entire feature groups were omitted during PRD generation. Validate by comparing BRD master feature inventory against PRD feature map.
9. **Orphan PRD features**: PRD includes features or user stories with no traceability back to BRD requirements — indicates scope creep or missing BRD documentation. Every PRD element must trace to at least one BRD requirement.

### Orphan Story Decision Tree

> Copy the following decision tree verbatim into the spawn prompt — do not paraphrase or summarize.

When an orphan story (US-xxx with no FR-xxx source) is found:
a. **Check indirect mapping**: Does it implement a BRD requirement indirectly (via NFR, business rule, or integration requirement)? If yes → document the indirect mapping with a note: "Implements [NFR/BRL/IR]-xxx indirectly." Flag in the validation report as "BRD gap — recommend adding FR-xxx in next BRD revision." Do NOT create the FR retroactively during Teammate 1's validation pass — flag it as "BRD gap — recommend adding FR-xxx" instead. The lead evaluates all BRD gap flags during Phase 4 and adds FRs to the BRD as part of the fix step (Phase 4, step 3) if the gap is CRITICAL or HIGH severity.
b. **Check product refinement or valid inference**: Is it a UX implementation of a documented BRD non-functional or business rule (e.g., empty state UI implementing a UI/UX requirement), OR a best-practice pattern required by the chosen tech stack (e.g., onboarding flows required by accessibility standards)? If yes → preserve the story (do NOT remove it or flag it as an orphan for deletion). However, if the story adds functional capability beyond UX polish, additionally flag it as a "BRD gap — recommend adding FR-xxx" to track the BRD improvement. Label the story as "PRD refinement" or "Valid inference — adds context clarity."
c. **Flag as scope creep**: If neither (a) nor (b) applies → flag for removal or explicit approval. Do NOT silently preserve untraced stories.

3. BUSINESS REQUIREMENT COMPLETENESS: For every business requirement (BR-xxx):
   - Verify at least one FR implements it.
   - Verify the PRD's product-level KPIs/OKRs align with the BRD's business-level KPIs.
   - Flag any BR that seems unaddressed.

4. NON-FUNCTIONAL REQUIREMENT COVERAGE: For every BRD NFR (NFR-xxx):
   - Verify the PRD's non-functional specifications (15-non-functional-specifications.md) include a corresponding product-level target.
   - Verify the target is more specific than the BRD requirement (e.g., BRD says "high availability" → PRD should say "99.9% uptime with < 5min MTTR").
   - Flag any NFR without a concrete PRD target.

5. SECURITY AND COMPLIANCE COVERAGE:
   - For every BRD security requirement (SR-xxx), verify a corresponding PRD specification exists.
   - For every BRD compliance requirement (CR-xxx), verify the PRD addresses it.
   - Flag gaps.

6. ASSUMPTION VOLUME RE-CHECK: Read `./plancasting/brd/_review-log.md` (if it exists).
   - Check if the Stage 1 review flagged the assumption percentage as CRITICAL (>=30%).
   - If flagged: verify in the validation report that the operator has reviewed and confirmed the assumptions before proceeding. If assumptions were not reviewed, add a CRITICAL finding: "BRD assumption volume exceeds 30% and has not been reviewed by the operator."

Output your traceability matrix to `./plancasting/_audits/spec-validation/traceability-by-coverage.md` and a gap report to `./plancasting/_audits/spec-validation/gaps-by-coverage.md`. The lead will consolidate these in Phase 4.

### Phase 4: Fix Specifications & Generate Report

After all teammates complete:

1. **Cross-feature consistency checks** (ALWAYS performed — whether validation was monolithic or split by feature group). Leverage Teammate 2's consistency analysis output rather than rebuilding from scratch:
   1. List all shared entity types (User, Organization, Project, etc.).
   2. For each entity, extract all field references from every feature group's specs — compare field lists, flag differences.
   3. Verify terminology is used consistently across feature groups (e.g., the same entity is not called "project" in one feature and "workspace" in another).
   4. Verify API endpoint naming conventions are consistent across all features (pluralization, casing, versioning).
   5. For workflows that span features, trace the data flow and ensure schemas match at each handoff point.

2. **Prioritize findings**: Categorize all issues as:
   - **CRITICAL**: Must fix before proceeding (blocks downstream stages) — missing requirements, contradictions, untestable criteria
   - **HIGH**: Should fix before proceeding (significant spec quality issue) — terminology inconsistencies, incomplete error states
   - **MEDIUM**: Can fix during validation (cosmetic or minor inconsistency) — style inconsistencies, redundant descriptions
   - **LOW**: Polish issue — formatting, minor wording, optimization suggestions that don't affect correctness

3. **Fix CRITICAL and HIGH issues directly**:
   - Update BRD files to fix BRD-side issues.
   - Update PRD files to fix PRD-side issues.
   - Add missing user stories, screen specs, or API specs where coverage gaps were found.
   - Fix terminology inconsistencies (choose one term, update both documents).
   - When adding new PRD artifacts (US, SC, API specs), allocate IDs from the next available number in the corresponding `_context.md` ID registry. Update the registry to reflect the new allocation. Note: Adding a missing US/SC for an existing FR is gap-closing (allowed). Adding entirely new FRs or BRs that are not in the original Business Plan is new feature addition (prohibited by Critical Rule 3).
   - Mark all fixes with `> ✓ VALIDATED [YYYY-MM-DD]: [fix description]` where YYYY-MM-DD is the current date in ISO 8601 format (e.g., `2026-03-14`). Obtain the current date via shell command `date +%Y-%m-%d`.
   - **Re-validate after fixes**: After fixing any BRD file, perform a targeted re-validation of ONLY the PRD files that reference the changed requirement. After fixing any PRD file, verify cross-references from other PRD files still resolve. This prevents cascade failures where fixing one issue silently breaks another. After modifying any BRD file, verify that no other BRD files reference the modified content with now-stale terminology or cross-references. Check `./plancasting/brd/_context.md` master inventory for consistency with any modified FR/BR/NFR IDs.
   - **Iteration limit**: After fixing a CRITICAL or HIGH issue, perform ONE re-validation of only the files modified. If new CRITICAL issues appear during re-validation, document them and escalate to lead — do NOT automatically fix and re-validate again. Limit to one fix-then-validate cycle per issue. **Escalation resolution**: The lead has two options: (a) manually fix the new CRITICAL issues and document fixes in the report, or (b) flag the issue as an unresolved CRITICAL finding in the report — this will cause the gate decision tree to evaluate as FAIL per rule 1. If re-validation discovers a NEW CRITICAL issue not in the original findings, the lead documents it in the report as 'Discovered During Fix Pass' and includes it in the gate decision. The lead does NOT start another fix cycle — new issues are documented for operator awareness.

4. **Consolidate teammate outputs**: Read all teammate output files (`traceability-by-coverage.md`, `gaps-by-coverage.md`, `consistency-report.md`, `quality-report.md`) from `./plancasting/_audits/spec-validation/`. Merge findings into a unified prioritized issue list, removing duplicates (same issue reported by multiple validators — keep the most detailed version).

5. **Generate `./plancasting/_audits/spec-validation/report.md`**:
   - Executive summary: overall spec health score (qualitative, based on severity distribution: **Excellent** = 0 CRITICAL, 0 HIGH, ≤ 5 MEDIUM; **Good** = 0 CRITICAL, ≤ 3 HIGH, ≤ 10 MEDIUM; **Fair** = 0 CRITICAL, ≤ 5 HIGH; **Poor** = any CRITICAL remaining OR >5 HIGH issues)
   - Requirements traceability matrix (complete, with coverage percentage)
   - Issues found by category and severity
   - Issues fixed during this validation
   - Remaining minor issues (documented but not fixed)
   - BRD ↔ PRD consistency score
   - Acceptance criteria quality score
   - Specification completeness score
   - Recommendations for the development team
   Note: The health score is a qualitative summary for human review. The gate decision (PASS/CONDITIONAL PASS/FAIL) below is the binding determination for pipeline progression.

6. **Update version history** in BRD and PRD cover pages to reflect validation changes. Note: BRD has `00-cover-and-metadata.md` for version history. PRD does not have a dedicated cover page — add a version note to `01-product-overview.md` instead.

7. **Stage 2B Gate Decision**: After all fixes are applied, determine the validation outcome:

   **Quick Decision Tree** (evaluate top to bottom, take the first match):

   > **IMPORTANT — Rule evaluation order**: Rules 1–4 are **blocking gates** evaluated first. Rules 5–8 are evaluated ONLY if rules 1–4 all pass. Do not skip to rule 5+ if any of rules 1–4 match.

   1. Any unresolved CRITICAL issues? → FAIL
   2. BRD assumption volume ≥ 30% AND operator has NOT confirmed review? Check `./plancasting/brd/_review-log.md` for an "Assumption Review Status" section with `Operator reviewed: YES`. If this marker is missing or set to `NO` → FAIL (the gate checks review STATUS, not assumption existence — if assumptions were reviewed and confirmed, rule 7 applies instead). Remediation: remediate Business Plan, re-run Stage 1, then re-run 2B.
   3. P0 feature coverage < 95%? → FAIL — P0 coverage is an independent gate checked separately from overall coverage.
   4. Overall coverage < 90%? → FAIL
   5. CRITICAL = 0 AND HIGH = 0 AND overall coverage ≥ 95%? → PASS
   6. CRITICAL = 0 AND HIGH ≤ 3 AND no P0 blockers AND overall coverage ≥ 90% AND (assumption volume < 30% OR operator confirmed)? → CONDITIONAL PASS (document each HIGH issue with root cause + remediation plan; if HIGH = 0, the coverage gap between 90–94% is the documented exception). Note: P0 ≥ 95% is guaranteed by this point because rules 3–4 are evaluated first (priority order) and catch P0 < 95% as FAIL.
   7. BRD assumption volume ≥ 30% AND HIGH ≤ 3 AND operator confirmed assumptions (`_review-log.md` contains `Operator reviewed: YES`)? → CONDITIONAL PASS (with caveat: "BRD assumptions exceed 30% — operator has reviewed and confirmed")
   8. No rule matched? → FAIL (edge case — e.g., CRITICAL=0 but HIGH>3 with ≥95% coverage, or other metric combinations not covered above. Document the specific combination of metrics and escalate to operator for manual gate decision)

   **Coverage definitions** (note: "coverage" in this stage means BRD→PRD requirement traceability — each FR has US + SC + API. This differs from Stage 3's "scaffold coverage" which measures PRD→file mapping):
   - **Complete coverage** for an FR = FR has at least one US AND at least one SC AND (at least one API endpoint/query/mutation function OR documented as frontend-only without backend interaction). For BaaS architectures (Convex, Supabase, Firebase), query/mutation functions count as "API" — adapt per `tech-stack.md`.
   - **Overall coverage** = (FRs with complete coverage / total FRs) × 100%
   - **P0 coverage** = coverage calculated only for FRs mapped to P0 features
   - P0 coverage and overall coverage are **independent requirements** — both must meet their thresholds

   - **PASS**: CRITICAL = 0, HIGH = 0, zero blockers remain, overall coverage ≥ 95% → proceed to Stage 3
   - **CONDITIONAL PASS**: CRITICAL = 0, HIGH ≤ 3 (none P0-blocking), each documented with workaround (root cause + remediation plan + acceptance criteria), P0 coverage ≥ 95%, overall coverage ≥ 90% → proceed to Stage 3 with documented exceptions.

     **Edge case**: If P0 feature coverage is below 95% but overall coverage is above 95%, the result is FAIL (not CONDITIONAL PASS) — P0 coverage is the binding constraint. P1-P3 coverage cannot compensate for P0 gaps.

     **P0 Coverage Calculation**: Read `./plancasting/prd/02-feature-map-and-prioritization.md` for the feature-to-priority mapping. For each P0 feature, identify its related FR-xxx IDs from the Feature Decomposition Map. Verify each of those FRs has coverage (US + SC + API). This cross-reference is necessary because the BRD uses MoSCoW priorities while the PRD uses P0-P3.

     **MoSCoW → P0-P3 Mapping**: Must Have requirements with high dependency chains (required by 3+ other features) map to P0. Must Have requirements with low dependency chains map to P0 or P1 (based on user impact). Should Have requirements map to P1 or P2. Could Have requirements map to P2 or P3. Won't Have requirements are excluded from the PRD scope. When a single BRD requirement spans multiple priority levels (e.g., core Must Have + enhancement Could Have), split into separate PRD features with independent priorities.

     **P0-Blocking Definition**: An issue is P0-blocking if it prevents a P0 feature from being fully coded and tested. Examples: missing core data entities, undefined authentication schema, broken API contracts for P0 features. Non-examples: documentation gaps, nice-to-have screen specs, performance optimizations.
   - **FAIL**: CRITICAL issues remain unresolved OR requirements coverage < 90% → STOP. Remediate and re-run Stage 2B

   **Blocker definition**: A blocker prevents proceeding to Stage 3. Always blockers: unresolved CRITICAL issues, requirements coverage < 90%, data model missing core entities referenced by multiple features, authentication schema undefined. Not blockers (fixable during coding): documentation gaps, screen spec formatting issues, minor terminology inconsistencies.

   Include the gate decision in the report: "Stage 2B Outcome: [PASS | CONDITIONAL PASS | FAIL]"

8. Output summary: total issues found, blockers fixed, coverage percentage, quality scores.

## Critical Rules

1. NEVER mark an issue as resolved without actually editing the source file — verify the fix is saved.
2. Default rule: Strengthen the PRD to match the BRD — do not weaken BRD requirements. Exception: When the BRD contradicts the Business Plan, fix the BRD to align with the Business Plan's intent (this is a correction, not a new feature). Then update the PRD to match the corrected BRD. If the BRD requirement is technically infeasible with the chosen tech stack, document the assumption with `> ⚠️ ASSUMPTION:` marker, adjust the PRD, and escalate to the lead for a decision.
3. NEVER add new features during validation — only close gaps for existing requirements.
4. ALWAYS preserve existing traceability links when editing files.
5. ALWAYS re-validate mermaid syntax after editing diagram sections. Teammate 3 validates syntax during initial review (Phase 2); the lead re-validates after fixes (Phase 4).
6. ALWAYS read CLAUDE.md before starting (all teammates, not just the lead).
7. When fixing a BRD file, check whether any PRD file references the changed requirement and update the PRD accordingly.
8. The `> ✓ VALIDATED` marker must include the specific change made, not just "fixed."
9. NEVER delete content without checking if downstream files depend on it.
10. If the combined BRD+PRD exceeds context limits (unlikely with the pipeline model's context window — see Token Budget section), split validation by feature group rather than by validation type.
11. ALWAYS use lowercase anchors in all cross-reference links when editing or creating new file sections (e.g., `#br-001` not `#BR-001`) to maintain compatibility with case-sensitive systems and downstream validation tools.
12. **Quantification Rule**: PRD should NEVER weaken BRD specifications. If BRD specifies quantified targets, PRD must preserve or enhance them. Any regression from quantified to vague (e.g., "99.9% uptime" → "high availability") must be flagged and fixed. This rule applies to all teammates during validation.

- BRD '99.9% uptime' → PRD 'high availability' = **REGRESSION** (fix by restoring quantification)
- BRD '99.9% uptime' → PRD '99.9% uptime with <5min MTTR' = **ENHANCEMENT** (correct)
- BRD 'response time < 200ms' → PRD 'fast response time' = **REGRESSION** (fix by restoring metric)
- BRD 'response time < 200ms' → PRD 'response time < 200ms at p95' = **ENHANCEMENT** (correct)
